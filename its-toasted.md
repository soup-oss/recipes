# Skill: It's Toasted

**CRITICAL:** This skill must remain permanently non-branded. It is a pure logical framework for strategic analysis. Use industry-neutral terminology unless referencing historical examples for context.

## 1. The Core Philosophy

The "It's Toasted" framework is based on the insight that market dominance is often achieved not by inventing a new utility, but by claiming a standard, industry-wide reality as a proprietary strength.

**The Historical Example:**
In the 1960s, a tobacco company (Lucky Strike) faced a market where every manufacturer's product was essentially identical. By identifying a mundane, universal step in production—toasting the tobacco—and branding it ("It's Toasted"), they claimed the narrative. Every other competitor toasted their tobacco too, but Lucky Strike was the only one that *said* so. 

This skill teaches the agent to find the "toasting" in a business idea and build a structural scaffold around it.

## 2. Pre-flight Check (Prerequisites)

**CRITICAL:** Before running this skill, the agent must firmly suggest to the user that they provide or point to structured, detailed documentation of what the product/business is in the first place. (garbage in garbage out)

Running this analysis in an empty directory or without a solid baseline of technical/functional specs will result in shallow, generic "hallucinated" strategies. The "It's Toasted" claim requires deep context to identify the specific mundane process that can be claimed as a strength. 

**Note to Agent:** You can use the format and structure of this document to help the user create such detailed documentation by dialoguing and refining ideas before starting the analysis phase.

## 3. The Analysis Scaffold (Output Structure)

When this skill is invoked, the agent must guide the user to populate a `docs/business/` directory with the following five files:

1.  **`thesis.md` (The Bet):** Defines the core philosophical disagreement with the status quo. What do we believe that everyone else treats as an optional or "probabilistic" side-effect?
2.  **`economics.md` (The Value):** Defines the mechanics of value capture—tiers, retention, and the "unit of work" that makes the model sustainable.
3.  **`landscape.md` (The Context):** A map of the current market, specifically identifying where competitors use "fuzzy" or "best-effort" language to describe the process we are claiming as "Deterministic."
4.  **`edge.md` (The "Toasted" Differentiators):** Cataloging the 3-5 standard realities we have turned into mandatory, visible, and proprietary truths.
5.  **`plan.md` (The Execution):** The adoption flywheel. How the product naturally creates its own sales channel through its core utility.

## 4. The Execution Workflow

### Phase A: Extraction (Interrogation)
The agent must not begin research until it has asked the user:
*   **The Commodity:** "What is the 'tobacco' of your industry? (The utility everyone provides)."
*   **The Process:** "What is the standard, boring way you handle that utility? What part of it are you currently treating as 'just how it works'?"
*   **The Mandatory Shift:** "What do competitors treat as optional metadata or 'nice-to-haves' that we should make a hard system requirement?"

### Phase B: Competitive Reconnaissance (Research)
The agent must use the web search tool to:
1.  Identify the top 3-5 competitors in the space.
2.  Analyze their documentation for "Fuzzy Language." Look for: *automatic*, *smart*, *best-effort*, *optional*, *flexible*, or *probabilistic*.
3.  Find the "apology": What do their users complain about in forums or issues that stems from a lack of deterministic structure?

### Phase C: Synthesis (The Claim)
The agent drafts the five files, ensuring every "feature" is reframed as a "Narrative Truth." 
*   *Bad:* "We have a log file."
*   *Good (Toasted):* "We provide a Mandatory Cognitive Audit Trail."

### Phase D: The Refinement Filter
The agent presents the `edge.md` and asks: **"If a competitor claimed they do this too, would they have to fundamentally break their current architecture to make it as Mandatory/Deterministic as ours?"**

## 5. Operational Guidelines

*   **Logic over Fluff:** Avoid marketing adjectives. Focus on the structural "Why."
*   **Presence over Permission:** Prefer differentiators that are triggered by system state or user presence rather than static configurations.
*   **Deterministic over Probabilistic:** Always lean toward binary outcomes (Success/Error) over "ranked" or "similarity" scores.
